Design Decisions

EKS Node Group Architecture

Three dedicated node groups prevent expensive GPU nodes from being consumed by commodity workloads.

GroupInstancePurposeTaint / Label
systemt3.medium ×2CoreDNS, kube-proxy, VPC CNI — Kubernetes overheadCriticalAddonsOnly
generalm7i.4xlarge ×2–8KEDA, Prometheus, Grafana, AIBrix control plane, ESOuntainted
gpug6e.12xlarge ×1–4vLLM inference pods onlynvidia.com/gpu=present:NoSchedule

Why g6e.12xlarge: 4× NVIDIA L40S gives 192 GB VRAM total — enough for Qwen2.5-14B at FP16 with tensor_parallel_size=2 (consuming ~56 GB across 2 GPUs, 136 GB free for KV-cache growth) or Qwen2.5-72B at FP8 with tensor_parallel_size=4. The L40S also supports NVLink-equivalent peer-to-peer bandwidth over PCIe Gen5, making TP=2 practical without NVLink fabric. A g5.2xlarge (A10G, 24 GB) would require INT8 quantisation for 14B, introducing measurable quality regression — unacceptable for compliance Q&A and financial advisory use cases.

Autoscaling Strategy

KEDA scales on two vLLM-native Prometheus metrics rather than CPU/memory, because GPU-bound inference saturates the GPU — and therefore KV-cache — well before CPU pressure is visible:

  • Queue depth trigger: vllm:num_requests_waiting / num_replicas > 5 — signals the serving fleet is falling behind request arrival rate
  • KV-cache trigger: vllm:gpu_cache_usage_perc > 80% — signals imminent eviction-induced latency degradation

A fallback HPA on CPU 70% handles pathological cases. The 300-second cooldown is intentional: at $16.29/hr per g6e.12xlarge, holding one idle node for 5 minutes costs $1.36 — cheap insurance against the 2–5 minute model load time a premature scale-down would impose on the next traffic spike.

02

GPU Cluster Management at Scale (10–50 nodes)

Node Lifecycle with Karpenter

At 10+ GPU nodes, replace EKS managed node groups with Karpenter. Karpenter is GPU-taint-aware, bins-pack model pods across instance types, and provisions nodes in ~90 seconds vs ~5 minutes for ASG warm-up. Key config:

  • NodePool specifying g6e.{xlarge,2xlarge,12xlarge} — allows Karpenter to right-size per request batch size
  • Consolidation policy: WhenEmptyOrUnderutilized — reclaims idle GPU nodes on a 5-minute drain window, blocking on Pod Disruption Budget (minAvailable: 1)
  • Drift detection: Karpenter replaces nodes whose AMI drifts from the NodeClass spec (e.g., new NVIDIA driver release), enabling zero-touch driver updates

Multi-AZ Topology

GPU nodes span all 3 AZs. Model pods use topologySpreadConstraints with maxSkew: 1 to prevent all replicas landing in one AZ. Key constraint: a serving pod cannot live-migrate (VRAM is not network-moveable), so AZ failure means losing those replicas until Karpenter provisions replacements — target RTO is <3 minutes for a single AZ loss at 2+ replicas.

Spot Instance Strategy

On-Demand (real-time traffic)

  • Minimum 1 pod always on On-Demand
  • Guaranteed capacity for synchronous user requests
  • PDB minAvailable: 1 blocks eviction

Spot (burst + batch)

  • KEDA burst replicas land on Spot (60–70% cheaper)
  • Batch inference jobs (vllm-batch namespace) exclusively Spot
  • Spot interruption handler drains pod gracefully on 2-min notice

GPU Health Monitoring

DCGM Exporter runs as a DaemonSet on all GPU nodes, exposing metrics to Prometheus:

  • XID error alerts: DCGM_FI_DEV_XID_ERRORS > 0 → PagerDuty + automatic node cordon (hardware fault indicator)
  • Temperature: DCGM_FI_DEV_GPU_TEMP > 85°C → throttle warning
  • ECC memory errors: uncorrectable ECC errors trigger node drain and replacement request via Karpenter disruption API
  • Node Feature Discovery labels each node with GPU model, VRAM, CUDA version — schedulers can filter on exact GPU capabilities

04

Data Residency & Compliance (UAE / CBUAE)

UAE Region Enforcement

All infrastructure deploys to me-central-1 (AWS UAE region, launched 2022 with in-country data residency guarantees). Enforcement layers:

  • SCP (Service Control Policy): deny ec2:RunInstances, s3:CreateBucket, eks:CreateCluster outside me-central-1 — no operator mistake can leak compute outside UAE
  • Terraform: provider "aws" { region = "me-central-1" } with CI validation that no resource block specifies a different region
  • S3 model weights bucket: object lock + replication disabled; no cross-region replication permitted for model weights containing training data residuals

Network Isolation

  • All GPU and general nodes in private subnets with no internet route; no IGW attachment on those route tables
  • VPC endpoints for ECR API/DKR, S3, Secrets Manager, CloudWatch — all traffic stays within the AWS network
  • Istio outboundTrafficPolicy: REGISTRY_ONLY in production — pods cannot initiate egress to undeclared hosts (prevents model from exfiltrating data via prompt injection)
  • ALB is the sole internet-facing endpoint; security group allows only 443 inbound from approved IP ranges (corporate + customer-facing API gateway CIDR)

Audit Logging for CBUAE

The Central Bank UAE (CBUAE) IT Governance Framework requires end-to-end traceability of all decisions made using AI systems in financial services:

  • Request/response logs: vLLM structured logging → CloudWatch Logs group with resource policy denying logs:DeleteLogGroup and logs:DeleteLogStream — immutable audit trail
  • User attribution: upstream API gateway injects X-User-ID and X-Session-ID headers; vLLM logs capture them per request
  • Retention: 7 years (matching MAS TRM and CBUAE IT governance expectations for AI decision records); enforced via CloudWatch retention policy + S3 Glacier export for years 2–7
  • CloudTrail: all AWS API calls, including model weight S3 access, in an organisation-level trail with integrity validation enabled
  • KMS audit: every decrypt of model weights or inference secrets appears in CloudTrail kms:Decrypt events — full key usage audit without application changes

100× Scale — What Breaks & What Needs Rethinking

Honest framing: 100× overnight is not just a software autoscaling problem. GPU capacity in a single AWS region is physically finite. You cannot reliably procure 100 g6e.12xlarge nodes on short notice — reserved capacity, procurement planning, and serving architecture changes are all required. Each layer is addressed below.

Layer 1 — vLLM Is Not Enough at This Scale; AIBrix Becomes Necessary

A single vLLM process on one g6e.12xlarge handles roughly 50–120 concurrent requests depending on context length and KV-cache state. At 100× baseline traffic that ceiling is hit immediately. The problems compound:

  • No cross-pod KV-cache awareness: two vLLM pods receiving the same system prompt both compute and store the same KV-cache independently, wasting VRAM and compute. At scale, prefix caching hit rate collapses.
  • No disaggregated prefill/decode: long-context financial document analysis (annual report ingestion, contract review) spends 80% of GPU time on the prefill phase — the same GPUs sit idle during the faster decode phase. Mixing both on the same fleet creates GPU utilisation valleys.
  • No intelligent routing: bare KEDA + multiple vLLM pods don't route requests to the pod with the best KV-cache match for that prompt prefix.

AIBrix closes these gaps: its EnvoyProxy gateway layer performs prefix-aware routing (send requests to the pod most likely to have a cache hit), disaggregates prefill pods from decode pods (each scaled independently), and provides a metadata service for cross-pod KV-cache state. At 10× scale AIBrix is optional; at 100× it's the difference between needing 100 nodes and needing 40.

The practical answer at 100× is a warm pool: maintain N nodes in a "ready but idle" state with model weights already loaded on an EFS PVC. The first burst request finds a warm pod in <15 seconds instead of waiting 6+ minutes. The warm pool size is determined by peak traffic history × ramp-up rate — typically 20–30% of peak capacity.

Layer 3 — GPU Procurement Is a Hard Physical Constraint

AWS's me-central-1 (UAE) is a smaller region with materially lower GPU instance availability than us-east-1. You cannot assume 50–100 g6e.12xlarge instances are available on-demand at any moment.

Accept that P99 spikes either queue (preferred) or spill to managed API (Bedrock fallback) with a PII scrubbing proxy — this hybrid mode keeps CBUAE compliance intact even when self-hosted capacity is exhausted.

Cost at 100× — The Number That Matters

Current baseline: 2 g6e.12xlarge On-Demand = $32.58/hr. At 100× traffic with efficient AIBrix prefix caching and disaggregation, you might need 20–30 nodes (not 200), because cache hit rates of 60–80% are achievable for financial chatbot workloads with repetitive system prompts.
