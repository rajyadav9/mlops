Design Decisions — Node Groups, vLLM Settings & Autoscaling

Why three EKS node groups

A single node pool collapses cost isolation. GPU nodes at $16.29/hr cannot be scheduled with commodity monitoring or control-plane workloads — one rogue Prometheus OOM will evict a model pod and the GPU sits idle while re-loading a 28 GB model for 3–5 minutes. Three tiers with hard taints enforce this separation at the scheduler level.

GroupInstanceTaint / SchedulingWorkloads
systemt3.medium ×2CriticalAddonsOnly:NoScheduleCoreDNS, kube-proxy, VPC CNI
generalm7i.4xlarge ×2–8untaintedKEDA, Prometheus, Grafana, Loki, AIBrix control plane, External Secrets
gpug6e.12xlarge ×1–4nvidia.com/gpu=present:NoSchedulevLLM inference pods only

Why g6e.12xlarge specifically: 4× NVIDIA L40S gives 192 GB VRAM total. Qwen2.5-14B in FP16 needs ~28 GB for weights plus KV-cache headroom — a single L40S (48 GB) is viable but leaves only 20 GB for KV-cache at 32K context, which fills under moderate load and triggers evictions (visible latency spikes). Running TP=2 across two L40S (96 GB available for one replica) gives comfortable headroom and upgrades cleanly to 72B via TP=4 across all four GPUs without changing the node type.

vLLM serving settings and the reasoning behind them

ParameterValueTrade-off made
--tensor-parallel-size214B fits on 1 GPU but leaves no KV-cache room. TP=2 doubles effective cache space; keeps the other 2 GPUs available for a second replica or model upgrade. Trade-off: intra-node NCCL overhead adds ~5–8 ms to TTFT.
--max-model-len32 768Covers 90th-percentile financial document lengths (annual reports, contracts). 64K is possible but doubles KV-cache VRAM consumption, reducing max concurrency. Chose 32K as the break-even point.
--gpu-memory-utilization0.9010% headroom for CUDA kernels and driver buffers. Setting 0.95 recovered ~9 GB VRAM but caused intermittent OOM during prompt spikes — not acceptable for a financial chatbot with strict SLAs.
Deployment strategyRecreateTwo model pods cannot share one GPU; a rolling update would schedule the new pod before the old one terminates, causing an OOM-kill of the running inference. Recreate causes a brief outage (~30 s) but is safer and predictable.
--enable-loraonServes multiple fine-tuned adapters (compliance Q&A, customer service) from one vLLM process without separate GPU allocations. Trade-off: adapter switching adds 1–3 s latency per swap; acceptable for request-level routing.

Autoscaling — why queue depth, not CPU

GPU-bound inference never saturates CPU meaningfully before the serving fleet falls behind. The real signal is whether requests are accumulating in vLLM's internal queue. Two KEDA Prometheus triggers:

  • Queue depth: vllm:num_requests_waiting / num_replicas > 5 — fleet is falling behind arrival rate
  • KV-cache pressure: vllm:gpu_cache_usage_perc > 80% — evictions imminent, latency about to spike

Fallback HPA on CPU 70% handles pathological cases (e.g., a bug causing busy-loop). Scale-down cooldown: 300 s. At $16.29/hr per node, holding one idle node for 5 min costs $1.36 — cheap insurance against the 3–5 minute model reload that follows a premature scale-down into a traffic spike.

GPU Cluster Management at Scale — 10 to 50 Nodes

Replace managed node groups with Karpenter at >5 nodes

EKS managed node groups use ASG warm-up (~5–8 min). At 10+ GPU nodes this is the dominant scaling latency. Karpenter provisions a g6e.12xlarge in ~90 seconds, is aware of GPU taints (only schedules GPU pods on GPU nodes), and right-sizes across instance families (g6e.xlarge / 2xlarge / 12xlarge) based on the actual resource request. Key Karpenter configuration:

  • Consolidation: WhenEmptyOrUnderutilized with 5-minute drain window — idle GPU nodes are reclaimed automatically, blocked by PodDisruptionBudget minAvailable: 1
  • Drift detection: Karpenter replaces nodes whose AMI drifts from the NodeClass (new NVIDIA driver release) — zero-touch driver updates with no manual cordon/drain cycle
  • Topology spread: model pods get topologySpreadConstraints maxSkew: 1 across AZs — prevents all replicas landing in one zone

Spot instance strategy across AZs

On-Demand — real-time serving

  • Minimum 1 pod always on On-Demand per model
  • PDB minAvailable: 1 blocks eviction
  • Karpenter base NodePool: capacity-type: on-demand
  • AZ-spread across all 3 private subnets

Spot — burst & batch

  • KEDA burst replicas land on Spot (60–70% cheaper)
  • Separate Karpenter NodePool: capacity-type: spot
  • Instance diversification: g6e.{xlarge,2xlarge,12xlarge} reduces interruption risk
  • Spot interruption handler gracefully drains pod on 2-min notice
  • Batch inference namespace exclusively Spot, scale-to-zero

GPU health monitoring

DCGM Exporter DaemonSet on all GPU nodes exposes hardware telemetry to Prometheus. Alerting rules:

DCGM_FI_DEV_XID_ERRORS> 0PagerDuty P1 + automatic node cordon — XID errors indicate hardware fault; model pods drain to healthy node
DCGM_FI_DEV_FB_USED (uncorrectable ECC)> 0Node drain + Karpenter disruption request for replacement — single-bit ECC is corrected, multi-bit is fatal to in-flight inference

Node Feature Discovery labels each node with GPU model, VRAM, CUDA version, and NVLink topology. Pod node selectors can target exact GPU capabilities — essential when running mixed g6e.xlarge (single L40S) and g6e.12xlarge (4× L40S) in the same cluster.

04

Data Residency & Compliance — UAE CBUAE

Region enforcement: me-central-1 with SCP backstop

All infrastructure targets me-central-1 (AWS UAE, launched 2022, contractually guarantees data remains in-country). A Service Control Policy at the AWS Organization root denies ec2:RunInstances, s3:CreateBucket, eks:CreateCluster, and rds:CreateDBInstance outside me-central-1. No IAM permission can override an SCP — this is a hard physical enforcement, not an honour system. S3 model-weight buckets have replication disabled and Object Lock enabled; weights cannot leave the UAE region even by accident.

Air-gapped network — three independent enforcement layers

The design philosophy is: each layer must independently prevent internet egress so that a failure at any one layer does not create an exfiltration path.

LayerMechanismIndependent protection
Infrastructure enable_nat_gateway = false, enable_public_subnets = false in Terraform. Private route tables have no default route — only VPC endpoint prefix entries. Packets to internet IPs are dropped at the routing layer; no forwarding path exists regardless of pod or application behaviour
Pod (Kubernetes) NetworkPolicy default-deny-all in every application namespace (vllm, monitoring, keda, aibrix-system). Explicit allow rules limit egress to DNS + VPC endpoint CIDR on 443 only. Even if a NAT GW is accidentally re-added, pods are blocked at the eBPF/iptables layer from initiating connections to unknown destinations
Application (Istio) outboundTrafficPolicy: REGISTRY_ONLY — only hosts with a declared ServiceEntry or VirtualService are reachable from any sidecar-proxied pod. Prompt injection attacks attempting to curl an attacker-controlled exfiltration endpoint are blocked by the Envoy sidecar, not just by routing rules

Full VPC endpoint set required: ECR (API + DKR), S3 (Gateway), Secrets Manager, KMS, STS, CloudWatch Logs, CloudWatch Metrics, SSM / SSMMessages / EC2Messages (no-SSH node access), EKS, EC2, Auto Scaling, ELB, SQS. Missing even one causes silent failures — kubelet cannot register, IRSA token refresh fails, or image pulls hang.

Audit logging for CBUAE AI governance framework

The CBUAE IT Governance Framework requires end-to-end traceability for AI-driven financial decisions:

  • Immutable request logs: vLLM structured logging → CloudWatch Logs group with a resource policy denying logs:DeleteLogGroup and logs:PutRetentionPolicy — no operator can shorten or delete the audit trail
  • User attribution: upstream API gateway injects X-User-ID, X-Correlation-ID; vLLM logs capture them per request; logs are non-repudiable
  • Retention: 7 years — CloudWatch (hot, year 1) → S3 Intelligent-Tiering → S3 Glacier Deep Archive (years 2–7). SCPs block deletion on the audit log account.
  • KMS CMK for all data at rest: EBS (model weights, ephemeral storage), EKS etcd secrets, ECR images, CloudWatch Logs — every kms:Decrypt event appears in CloudTrail with principal, timestamp, and resource ARN
  • PII gate (Day 0 requirement): a redaction proxy between the API gateway and vLLM detects and masks PII before prompts are logged — without this, audit logs become a PII breach
05

100× Scale — What Breaks First, What Changes

Key framing: 100× overnight is not a software autoscaling problem. GPU capacity in me-central-1 is physically finite. You cannot reliably procure 100 g6e.12xlarge nodes on short notice in a smaller AWS region. The solution requires architecture changes, serving efficiency improvements, and procurement planning in parallel.

Failure sequence — ordered by when each breaks

#What breaksRoot causeFix
1 KEDA ceiling (maxReplicaCount: 8) Hard config limit hit immediately at 100× — queue depth spikes, replicas saturate at 8, latency climbs without bound Remove the ceiling. Let GPU supply (account quota) be the natural limit, not a config value. Add SLO-based alerting so humans are notified before queue depth becomes user-visible.
2 vLLM single-process throughput ceiling One vLLM process handles ~50–120 concurrent requests depending on context length. Long financial documents (32K tokens) drop this to ~10–20. 100× traffic needs 100× more processes — but naive horizontal scaling doesn't help if they all independently compute the same prefix. AIBrix becomes mandatory here. Its EnvoyProxy gateway does prefix-aware routing: requests sharing a common system prompt are sent to the pod most likely to have that prefix cached. For a financial chatbot where 80% of requests share the same 2K-token system prompt, this achieves 60–80% KV-cache hit rates — effectively 3–5× throughput per node without adding hardware.
3 Synchronous request path ALB → vLLM direct with no buffering. A traffic spike that takes 5 minutes to absorb (node provisioning + model load) causes client timeouts and cascading retries that amplify the spike. Insert an SQS FIFO queue between the API gateway and vLLM. Decouple acceptance rate from serving rate. Clients receive a request ID immediately; results streamed via SSE or polled. Queue depth becomes the KEDA trigger, giving the cluster time to scale without dropping requests.
4 GPU node provisioning lag Even with Karpenter (~90 s), 20 new GPU nodes take 90 s (provision) + 5 min (model load) × however many run in parallel = 8–12 min to absorb the spike. Financial chatbots with SLA <2 s p99 cannot wait 12 minutes. Warm pool: keep N nodes running with model weights pre-loaded on EFS PVC — no download on scale-out. Warm pods are ready in <15 s. Size the pool to absorb the P95 ramp rate. Costs ~$16/hr per idle node but prevents latency SLO breach.
5 GPU procurement ceiling me-central-1 is a smaller AWS region. g6e.12xlarge default quota may be 8–16 vCPUs for new accounts. Even with quota increases, AWS cannot always fulfil 50+ GPU instance requests on-demand during high-demand periods. Three-tier capacity (see below). Additionally: AIBrix disaggregated prefill/decode separates compute-bound (prefill) from memory-bandwidth-bound (decode) phases — each scaled independently. This alone can reduce required node count by 30–40% vs running unified pods.
6 Observability stack 100 pods × vLLM high-cardinality metrics = millions of active Prometheus series → OOM. Loki single-distributor saturates at ~50 GB/hr ingestion. Thanos sharded query + Ruler for Prometheus. Loki distributed mode (already scaffolded in prod values — increase replicas). Recording rules pre-aggregate high-cardinality series before storage.
